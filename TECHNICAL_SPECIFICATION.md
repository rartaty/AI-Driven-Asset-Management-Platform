# 資産管理・自動運用システム 詳細技術仕様書 (Technical Specifications)

> **【リポジトリ公開に関するポリシー】**
> 本ドキュメントは、当システムのアーキテクチャ設計、品質保証基盤、およびデータモデリングのショーケースです。**具体的な投資アルゴリズム、閾値、およびAIのプロンプトエンジニアリング等のコアロジックは非公開**としています。本仕様書では「それらのロジックをシステム上でいかに安全かつ高速に実行・運用しているか」というエンジニアリングの側面に焦点を当てております。

---

## 1. システム・アーキテクチャ全体構成
本システムは、高い堅牢性と再現性を保つため、コンテナによるハイブリッド構成とテスタビリティを意識した設計を採用しています。

- **ホストOS環境**: Windows 11 / 10
- **コンテナエンジン**: Podman (またはDocker Desktop)
- **インフラ管理**: `docker-compose.yml`によるオーケストレーション

### 1.1 構成コンポーネント
1. **Frontend (Next.js コンテナ)**
   - ユーザー操作画面と描画を担当。
2. **Backend (Python 3.11 / FastAPI コンテナ)**
   - REST API サーバーおよび、スケジュールワーカー（`APScheduler`）の同居稼働によるバッチ処理を担う心臓部。
3. **Database (SQLite / SQLAlchemy 2.0)**
   - - アセット状態、AI履歴、意思決定ログの永続化。個人ローカル運用に最適化されたファイルベースDB（`data.db`）。将来的なクラウド移行を見据え、SQLAlchemy 2.0のORMを用いてRDBMS非依存な設計としています。

## 2. ソフトウェア・技術スタック詳細

### 2.1 バックエンド (Python 3.11)
実行時エラーを未然に防ぐため、厳格な型ヒント (`typing`モジュール) とPydanticによるバリデーションを義務付けたコーディングを採用しています。

- **Webフレームワーク**: FastAPI
- **データベース・ORM**: SQLAlchemy 2.0 (SQLインジェクション対策およびスキーマ管理)
- **データ分析・演算**: `pandas`, `numpy`, `scipy` による高速な行列演算と時系列データ処理
- **外部通信**: `requests`, `websocket-client` (証券APIとのストリーミング通信)
- **機密情報管理**: `boto3` を用いたAWS SSM Parameter StoreからのKMS暗号化APIキー動的取得

### 2.2 テスト・品質保証基盤 (Testing Strategy)
「金融システムとしての堅牢な振る舞い保証」を目的として、以下のアプローチでテスト基盤を組み込んでいます。

- **Unit Testing**: `pytest` を用いたビジネスロジック（リバランス計算、状態遷移など）のテスト。
- **Mocking**: `pytest-mock` を活用し、実際の証券APIへリクエストを飛ばさずに「想定されるRESTレスポンスJSON」を用いた結合テストを実施。
- **DI (Dependency Injection)**: FastAPIの `Depends` 機能を使い、テスト環境下では自動的に「APIクライアント」を「Mockクライアント」へ差し替えるアーキテクチャを採用。

## 3. ディレクトリ構成 (AI-Friendly Architecture)
保守性と単一責任の原則（SRP）を厳守した構成です。コアロジック部分は隔離して管理しています。
```text
/trading-system
├── docker-compose.yml       # コンテナ群の環境定義
├── .env                     # 環境変数 (Mock動作フラグ等 ※Git管理外)
├── /data                    # 永続化マウント (SQLite DB、Logs)
├── /frontend                # Next.js のフロントエンドソース
└── /backend                 # Python 3.11 バックエンド
    ├── pyproject.toml       # pytest設定、linter/formatter等定義
    └── /src
        ├── /api             # 各種外部APIリクエスト (Mock切り替え機構付き)
        ├── /core            # AWS SSM復号処理、Discord通知等の共通基盤
        ├── /models          # SQLAlchemyテーブル定義
        ├── /routers         # FastAPIのエンドポイント群
        ├── /strategy        # 【非公開】自動取引ロジック推論モデル群
        └── /runner          # メインジョブスケジューラ (本稼働用)
```

## 4. 取引・運用エンジンのアーキテクチャ

具体的なアルゴリズムは非公開ですが、各運用層がシステム内でどのようにデータ処理を行っているかを解説します。

### 4.1 長期運用（V6コア・サテライト）ロジック
財務指標とマクロ環境を統合評価するバッチ処理エンジンです。
- **データパイプライン**: EDINET等からXBRLパーシングにより財務データ（キャッシュフロー、EBITDA等）を自動抽出し、Pandas DataFrame上で独自のスコアリングモデルを実行します。
- **AI連携**: ローカルLLMを用いたマクロ環境評価を非同期バッチとして実行し、最終的な投資判断のスコア補正に使用しています。

### 4.2 短期運用（VWAP平均回帰）ロジックと技術スタック
市場構造の歪みを突く、高頻度なストリーミング解析エンジンです。
- **リアルタイム処理**: 証券会社のPush API（WebSocket）から受信した板情報・価格データを数ミリ秒単位で処理。
- **特徴量エンジニアリング**: フラクショナル微分によるデータ定常化や、オーダーブック・インバランスの算出により、ノイズを排除したシグナル検知を行っています。
- **異常検知の絶対化**: アクティブ時間帯（09:15〜14:30）は重量級のAI推論を完全にシャットダウンし、遅延（レイテンシ）を最小化する設計としています。

### 4.3 投資信託運用・リバランスロジック (Passive Core)
- **バッチ・スケジューリング**: 毎営業日の14:50（全決済完了後）にスケジューラから自律的に呼び出される「利益振替（Profit Sweep）」バッチ処理として実装。
- **動的リバランス**: 各資産クラスにおける独自の Tolerance Band（乖離許容幅）を監視し、閾値を超えた場合に自動でリバランス金額を算出するロジックを実装しています。

### 4.4 レポート自動生成エンジン (Report Generator)
- **処理プロセス**: 日次・月次・年次のタイミングで、DBからトレードログや損益データを集計。これらのログデータをローカルLLMに渡し、「If-Thenシナリオ（エントリー/エグジットの最適化案）」を生成させ、レポートファイルとして指定ディレクトリへ自動アーカイブする非同期パイプラインを構築しています。

## 5. アラート・監視仕様 (Discord)
運用中のフェイルセーフとトレーサビリティを確保するための設計です。

- **構造化エラーハンドリング**: 例外発生時は、タグ `[COMPONENT]` を先頭に付与して原因レイヤー（DB, Web, API）を明確化し、Discordへ即時通知します。
- **キルスイッチ連動**: 「資産の3%ドローダウン」などの致命的な事象を検知した場合、システム変数 `is_kill_switch_active = True` が発火し、コードレベルで完全な「新規発注の遮断」を実行。同時に緊急アラートを送出します。
- **トランザクション通知**: トレード終了時、実現損益データとともに、AIによる「定性分析結果（なぜその結果になったか）」を併せて通知します。

## 6. データベース構造 (Database Schema)
フロントエンドでの高度な可視化と、AIによる振り返り分析を支える主要なデータモデルです。

### 6.1 `Daily_Asset_Snapshot` (日次資産スナップショット)
多角的な時間軸分析（ポートフォリオ成長曲線など）の描画ソースとなるテーブルです。
- **主要カラム**: `date`, `bank_balance` (銀行残高), `buying_power` (買付余力), `trust_value`, `long_solid_value`, `short_term_market_value` など、各バケツの評価額を日次で記録。

### 6.2 `Trade_Tick_Log` (短期トレード用ログ)
イントラデイの分単位の損益推移や、AIのレジーム判定を可視化・分析するための時系列ログです。
- **主要カラム**: `timestamp`, `unrealized_pnl` (未実現損益), `realized_pnl` (実現損益), `regime_type` (AIによる相場環境の判定フラグ)。

### 6.3 `Report_Archive` (運用レポート・アルバム)
AIが自動生成した振り返りレポートを管理・表示するためのメタデータテーブルです。
- **主要カラム**: `report_id`, `report_type` (日次/月次/年次), `target_date`, `file_path` (レポート実体のパス), `ai_summary` (If-Then分析などのテキスト)。
