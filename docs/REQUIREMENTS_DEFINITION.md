# 資産管理・自動運用システム 統合要件定義書

提示いただいたフォーマットに基づき、本プロジェクトの全容と「非機能要件」をマッピングした統合ドキュメントです。

---

## 1. プロジェクト概要
銀行API（OpenCanvas）と証券API（三菱UFJ eスマート証券）を高度に連携させ、AI（Codestral/Gemma 2）による意思決定支援と資産の自動運用をシームレスに行う次世代型プラットフォーム。ローカルPC上（Podman）でのセキュアな稼働を前提とし、フロントエンドにNext.jsによるモダンなUIを提供する。

## 2. システム機能要件（Scope）
### 2.1 資産・取引管理 (Asset Management)
- **資産の可視化**: 銀行残高、買付余力、株式・投資信託の総額、割合、評価損益を「1スクロール」のダッシュボードで一覧表示。投資信託および長期運用株式については、具体的な保有銘柄ごとのポジション情報（銘柄名、保有数、取得単価、個別損益）も併せて表示する。
- **多角的な時間軸分析**: 日次（振替実績）、月次（ポートフォリオ成長曲線）、年次（年間利回り）の3つの時間軸での資産推移グラフ表示。
- **短期トレード・イントラデイ詳細**: 分単位での実現・未実現損益推移グラフ、VWAP乖離率、レジーム判定のオーバーレイ表示。
- **取引履歴と理由の提示**: 各取引に関する損益データ（金額・割合）と、AIがなぜその売買を行ったか（特に損失理由）のドリルダウン表示。
- **レポート・アルバム**: 日次・月次・年次で生成されるAIの振り返り（改善点・If-Then分析を含む）レポートをアーカイブ化し、カレンダー・リスト形式で閲覧可能にする。
- **情報照会**: ユーザー指定銘柄や取引中銘柄の個別指標データ取得。

### 2.3 セキュリティ・通知 (Security & Alert)
- **認証・アクセス制御 (Authentication & Access Control)**: 管理者権限（個人利用想定）へのアクセス制限。ローカルネットワーク稼働を前提とする。
- **機密情報の暗号化**: APIキーの直書きを禁止。AWS SSM Parameter Store (KMS暗号化) 等を利用し、実行時に動的に認証情報を取得。
- **スマート通知**: 約定通知、AI推奨銘柄検知、またはシステムエラー（APIタイムアウト、ロジック異常等）発生時に即時プッシュ通知を **Discord** に対して行う。異常検知時（ドローダウン等）はキルスイッチと連動。

## 3. アーキテクチャと技術スタック
- **インフラ**: Windows 11ホスト / Podman / docker-compose
- **データベース**: SQLite (`backend/src/data.db`、個人ローカル運用デフォルト) / SQLAlchemy 2.0 ORM (PostgreSQL 互換 SQL を生成)。クラウド化時は [DB_MIGRATION_GUIDE.md](DB_MIGRATION_GUIDE.md) で PostgreSQL へ移行可能
- **バックエンド API**: Python 3.11, FastAPI, pandas/numpy (分析ロジック)
- **フロントエンド UI**: Next.js, バニラCSS/CSS Modules (Tailwind不使用), ダークモード・グラスモーフィズムデザイン

## 5. 資金管理・リスク管理ルール
- **生活防衛費（積立金）ブロック**: 人間が設定した固定費にはAIはタッチできず、銀行口座 (OpenCanvas) は読取専用扱い。出金機能は意図的に使用せず、証券口座とは独立プールとして管理 (§2.2 参照)。
- **🌟 投資判断ファースト原則**: 動的トップレベル比率は **目標値 (aspirational)** であり、強制配分ルールではない。AIが利益見込みありと判断した場合のみ投資を実行する。**比率維持のためだけに無理な投資は禁止**。投資機会がなければ証券口座は現金100%保持も許容。詳細は §2.2 参照。
  - 長期: 世の中の経済状況 + マクロ市場環境 + 銘柄分析 + ロジック で判断
  - 短期: 銘柄分析 + ロジック で判断
- **VIXギアシステム**: VIX指数を監視し、20以下なら防御（キャッシュ温存）、35以上なら暴落時のリバウンド狙いとして資金をアクティブ化。**動的トップレベル比率の見直しトリガーとしても機能** (§2.2 ポートフォリオ見直しタイミング参照、四半期定期 + VIXギア発火 + 手動の3トリガー)。

## 6. 異常系・エッジケース対応設計 (Fail-safe)
- **資産ドローダウン検知 (キルスイッチ V2)**: ポートフォリオが短時間で閾値（例: 3%）減少した場合、新規買付注文を即座にブロックする (`is_kill_switch_active`)。詳細仕様は別文書 [docs/operations/KILL_SWITCH_V2_SPEC.md](operations/KILL_SWITCH_V2_SPEC.md) (作成予定) で管理:
  - **永続化**: DB (`User_Settings.is_kill_switch_active`) + 物理 `.kill.lock` ファイル (ホスト共有ボリューム上)。コンテナ再起動でもブロック状態を維持
  - **非対称オーバーライド**: 新規エントリーのみブロック、**既存ポジションの決済 (Exit) は許可** (ショート建玉の買戻し決済を許容するため必須)
  - **解除**: 自動解除しない。手動 (Discord WebHook / Next.js緊急停止ボタン経由) でのみ解除可能
- **エラーハンドリング**: キャッチしたExceptionにすべて`[COMPONENT]`を付与し、原因レイヤー（DB, Web, API）を明確化してDiscordへ通知。

## 7. 運用・監視ループ設計
- **バックグラウンドジョブ**: `APScheduler` または `schedule` モジュールにより時刻ベース（朝8:30のトークン取得、14:50の強制決済、年次の長期見直し）で自律稼働するループの構築。

## 8. データベース設計（主要エンティティ）
- `User_Settings`: 固定積立額・ドローダウン閾値・**`is_kill_switch_active`** などのコンフィグ
- `Cash_Pool_Status`: リアルタイムな銀行口座残高・証券口座買付余力 (両プール独立管理 §2.2)
- `Target_Portfolio` / `Market_Context`: **動的トップレベル目標比率** (四半期 + VIXで更新) と市況データ
- `Trade_Logs` / `System_Logs`: 発注意図、AI思考プロセス、システムエラーの永続化
- `Asset_Master`: 個別銘柄の長期コア・長期サテライト・短期区分マスタ。**`is_active=False` で上場廃止銘柄も保持** (Survivor Bias排除のためPoint-in-Time再現性を維持)
- `Daily_Asset_Snapshot`: 日次資産スナップショット (両プール残高・各 bucket 評価額・短期Sweep累計額)。詳細カラム定義は [docs/TECHNICAL_SPECIFICATION.md §6.1](TECHNICAL_SPECIFICATION.md) 参照
- `Financial_Reports`: XBRL由来の財務データ (FCF/EBITDA/総資産/発行済株式数)。JSONB活用で将来の科目追加に柔軟対応
- `Daily_Price_History`: 補正済み株価・出来高 (株式分割・併合の Adjusted Price 再計算済み)
- `Trade_Tick_Log`: 短期トレードのイントラデイ分単位損益・レジーム判定ログ
- `Report_Archive`: AIによる振り返り（If-Then分析等）を含む、自動生成レポートのメタデータと保存パス

# 非機能要件定義項目一覧（適正化版）

> **【エンタープライズ要素の適正化について】**
> 提示いただいた非機能要件は「巨大金融機関のデータセンター稼働」などを想定したものを含んでいます。「個人投資家のローカル運用システム」としては大規模すぎるもの（オーバースペック）につきましては、下記の方針で整理（N/A化、または縮小）いたしました。
> - **[A] 可用性:** DBバックアップやエラーリカバリは自律化するが、DRサイト（遠隔地の災害予備データセンター）構築等は廃止。
> - **[B] 性能・拡張性:** 数万の同時アクセスを想定した負荷分散要件は除外。ただしAI/VWAPアルゴリズム処理のレスポンスは高速化要件として残す。
> - **[C] 運用・保守:** 24時間/365日のベンダー保守・予備機材の規定等は除外。
> - **[D] 移行性:** データ移行ステップの複雑な要件は新規システムのため縮小。
> - **[E] セキュリティ:** ここが最重要。ローカル稼働でもAPI漏洩等を徹底して防ぐため、ほぼそのままの強度で適用。
> - **[F] エコロジー:** データセンターの床耐荷重や冷房制約などは除外。

### A 可用性（一部オーバースペック適正化）
- **A.1 継続性**: 単一PC上のコンテナ稼働であるため、RTO/RPOの保証は行わない。ただしプロセス異常終了時はDocker（Podman）の`restart: always`ポリシーで自動再開する構成とする。
- **A.2 耐障害性**: HWレイヤの冗長化（A.2.1〜A.2.5）は「対象外（ローカルPC依存）」。
- **A.2.6 データ**: SQLite データファイル (`backend/src/data.db`) の定期コピーによるバックアップを実施 (ファイルベースのため停止不要・無停止スナップショット可)。将来 PostgreSQL 移行時はコンテナデータの定期スナップショット (ホストディレクトリへのマウント) に置き換え。
- **A.3 災害対策（遠隔地保管など）**: 「対象外（個人利用につきオーバースペック）」。
- **A.4 回復性**: Discordのエラー通知をもとに、ユーザー（自身）が手動復旧を行う手順とする。

### B 性能・拡張性（個人スケールへ適正化）
- **B.1 業務処理量（B.1.1.1 ユーザ数 等）**: 「単一ユーザー・少数の端末アクセス」に限定。（数万の同時アクセス対応などは除外）
- **B.2 性能目標値**: 短期アルゴリズム（VWAP計算等）においては、kabuステーションPush APIから受信後「数ミリ秒以内」の意思決定と発注を目標とする（遅延の最小化）。
- **B.3 リソース拡張性**: コンテナ起動時のCPU/メモリ制限によるコントロール、必要に応じたクラウド拡張性（AWS移行）をコードの柔軟性として担保。
- **B.4 性能品質保証（HW専有等）**: 「対象外（ローカルPCスペックに依存）」。

### C 運用・保守性（ベンダー保守条項を除外）
- **C.1 通常運用**: SQLite ファイルバックアップ等の運用タスクはジョブスケジューラ (APScheduler) でスクリプト化 (例: 日次で `data.db` を `backups/data-{YYYYMMDD}.db` へコピー、世代管理付き)。
- **C.2 保守運用**: パッチ適用は手動、またはコンテナ再ビルドによって行う。
- **C.3, C.5 （保守ベンダー・サポート体制）**: セルフマネージド（自己責任体制）のため全て「対象外（オーバースペック）」。

### D 移行性（システム開発時の要件）
- 新規構築のため、レガシーシステムからのデータ移行要件（D.4, D.5等）の大半は「対象外」。ダミーデータを活用した単体・結合テストのサイクルのみ適応する。

### E セキュリティ（厳格適用）

#### E.1 前提条件・制約条件
- **E.1.1.1 順守すべき社内規程、ルール、法令、ガイドライン等**:
  - 国内の「金融商品取引法」に基づく投資助言・代理業等の規制抵触を避けるため、システムを他者へ公開・提供しない（完全個人利用）。
  - 各種金融機関（三菱UFJ eスマート証券、OpenCanvas等）が定める **API利用規約（スクレイピングの禁止、リクエスト頻度の制限等）** をシステム仕様の絶対的な上限として遵守する。

#### E.2 セキュリティリスク分析
- **E.2.1.1 リスク分析範囲**:
  - **保護対象資産**: データベース (SQLite ファイル `backend/src/data.db`) に保存される資産状況、実行環境上のAPIキー、AWS認証情報。
  - **脅威と対策範囲**: APIキーの平文保存による漏洩（AWS SSM/KMSによる動的取得で対策）、コンテナ外からの不正アクセス（ホストOSのファイアウォールとDockerネットワーク隔離で対策）。

#### E.3 セキュリティ診断
- **E.3.1.1〜E.3.1.3 ネットワーク・Web・DB診断**:
  - サードパーティ業者によるペネトレーションテスト等の外部診断は対象外とする。
  - **代替措置**: 開発時にコード解析ツール（Pythonの `bandit`, Node.jsの `npm audit` 等）を導入し、ソースコード診断と依存関係の脆弱性スキャンをセルフで実施する。

#### E.4 セキュリティリスク管理
- **E.4.1 セキュリティリスクの見直し**:
  - **見直し頻度**: 月次、または重大な脆弱性（CVEスコアHigh以上）のニュース発覚時。
- **E.4.2 セキュリティリスク対策の見直し**:
  - **運用開始後のリスク対応・方針**: 依存ライブラリに致命的な脆弱性が発見された場合、自動売買ジョブを即座に一時停止し、安全が確認されたバージョンへの引き上げを最優先する。
- **E.4.3 セキュリティパッチ適用**:
  - **適用範囲と方針**: ホストOS (Windows) のパッチはOSの自動更新に依存。コンテナ内部 (Python, Node) は、最新の公式軽量イメージへの再ビルドで適用する。SQLite はファイルベースのため対象外 (Python ランタイム同梱の `sqlite3` ライブラリで稼働)。
  - **適用タイミング**: 週末や祝日など、株式市場が閉まっている（システムがアクティブに動いていない）タイミングでの手動適用を原則とする。

#### E.5〜E.10 その他のセキュリティ要件
- **E.5.1 アクセス・利用制限**: DBレイヤーおよびWebUIに対する基本的なパスワード/トークン認証（ローカルといえど保護）。
- **E.6.1 データ暗号化**: **絶対必須要件**。平文のAPIキー埋め込みを禁止し、AWS SSM等を利用して鍵管理。
- **E.7 不正追跡・監視**: Discord通知および、`python-json-logger`を利用した全トラフィック・意思決定の可視化。
- **E.8〜E.10 ネットワーク・マルウェア**: ホストOS（Windows）自体のファイアウォール・アンチウイルスに依存。WAF等の導入は本番外部公開プロセスまで除外。

### F システム環境・エコロジー（除外）
- 床荷重、空調性能、CO2排出量、特定機器指定など、物理的データセンター向け要件は「すべて対象外（オーバースペック）」とする。
