## サマリ (250 words)

Phase 4 (`core/` 本実装) 着手直前のスナップショットとして、`docs/` 仕様 vs `backend/src/` 実装を 7 カテゴリで監査した。

**全体評価**: ハーネス9層完成と並行して `core/` 3 ファイル (aws_ssm / discord / logger) のスケルトンが既に配置済みで、ディレクトリ構造の Spec 準拠度は前回監査 (`api`/`core`/`models`/`routers`/`strategy` + `services`) より大幅改善。要件と実装の乖離の中心軸は Phase 4 で解消されるよう設計されている (3 ファイルとも `# TODO: Phase 4 で本実装` コメントあり)。

**最大のリスク領域**:
1. **Risk Management (high)** — `is_kill_switch_active` カラムは DB に存在するが、書き込み・参照・発注ブロック処理が一切未実装。生活防衛費保護・VIXギア発火判定も同様にゼロ。schema 上の placeholder のみ。
2. **Security (high)** — SSM/Discord/Logger 全て stub、`KABUCOM_API_PASSWORD` を `os.getenv` 直読み (要件 E.6.1 違反)、FastAPI ルーターに認証ゼロ (要件 E.5.1 違反)。
3. **Scheduler (med)** — 朝08:30 トークン取得・年次見直し・週次/日次リバランスのジョブ未登録。短期トレード時刻も 09:15 ではなく 09:00 始動。

**強み**:
- DB schema は仕様改訂 (`cumulative_sweep_to_long_solid` 等) を完全反映済み。3 層ポートフォリオ + 動的トップレベル比率関連テーブルも揃っている。
- Frontend は dashboard / reports の主要骨格が動作。

**結論**: Phase 4 で `core/` 3 件本実装 + Kill Switch ロジック配線 + scheduler ジョブ追加を併走すれば、要件適合度は実用ラインに到達する見込み。

## ❓ Ambiguous (人間判断要)

- **services/ ディレクトリ** — 仕様 §3 ディレクトリ図には未記載だが、`scheduler.py` `portfolio_sync.py` `paper_trader.py` `ai_analyzer.py` の置き場として実用上必要。命名は問題ないが、仕様書側に services/ を追加するか、core/runner/ に再配分するかの方針確認が必要。
- **Pydantic Response Model** — code-style.md は新規コードに Pydantic 使用必須としているが、ルーター層は素の dict で返している。Pydantic ResponseModel の段階的導入タイミングを Phase 4 に含めるか?
- **TEST_MODE vs TRADE_MODE 二重環境変数** — `KabucomAPIClient` は `TEST_MODE` ([kabucom.py:19](../../backend/src/api/kabucom.py#L19))、scheduler は `TRADE_MODE` ([scheduler.py:94](../../backend/src/services/scheduler.py#L94)) を参照。CLAUDE.md は `TRADE_MODE` のみを正としているため、`TEST_MODE` は廃止 or 統合方針確認が必要。
- **`paper_trader.py` の `Short` カテゴリ固定** — [paper_trader.py:83](../../backend/src/services/paper_trader.py#L83) で全ポジション `category="Short"` 固定。schema の 4 値 Enum と整合しない。PAPER モード時は long/short を分離記録すべきか方針確認。
- **Profit Sweep 「振替先 = 長期コア」改訂の浸透** — schema は対応済 (`cumulative_sweep_to_long_solid`) だが、calculate_profit_sweep の戻り値キー名 `sweep_to_trust` が旧命名のまま。リネームの優先度判断。
