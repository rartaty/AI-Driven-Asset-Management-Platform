# 資産管理・自動運用アプリケーション 要件および設計全体像

本ドキュメントは、システム全体の機能要件、アーキテクチャ、およびAIモデル連携プロセスを俯瞰・参照するためのマスタドキュメントです。実装の各フェーズにおいて、このドキュメントを随時参照し、ブレがないかを確認します。

## 1. システム全体像と目的
当システムは、銀行API（OpenCanvas）と証券API（三菱UFJ eスマート証券）を連携し、「資産管理」と「AIによる自動取引（短期・長期）」を完全統合した次世代型個人投資家向けアプリケーションです。

### 1.1 プロジェクト全体方針
- **資産管理**: UI（Next.jsダッシュボード）にて、銀行残高や株・投資信託のポートフォリオ状況を直感的に「1スクロール」で確認・管理できること。
- **自動運用**: 完全にルール化された投資ロジックとAI（Codestral/Gemma 2）を用いて、人間の感情を排除し長期的な資産増加を図る。

---

## 2. アーキテクチャおよび技術スタック

### 2.1 インフラと連携
- **実行環境**: Windows 10/11 上の Podman。`docker-compose` によってDBとアプリケーションを連携し管理。
- **データベース**: SQLite (`backend/src/data.db`、個人ローカル運用前提)。SQLAlchemy 2.0 ORM を用いたデータ中心管理。

### 2.2 ソフトウェア層
- **バックエンド (API & Logic)**: Python 3.11 + FastAPI。スケジュール実行、データ取得、リアルタイム通信、AI連携（ローカルOllama）を担う。
- **フロントエンド (UI)**: Next.js + CSS Modules (Tailwind非推奨)。ダークモード・グラスモーフィズム等の高品質デザイン。
- **通知および監視**: トレード結果やシステムエラーは、Discordへ即時WebHook通知。

## 3. 機能要件 (UI/UX)
ユーザーがダッシュボード上で実行・確認できる機能の一覧です。

1. **ルール遵守状況の確認**: AI/システムが定めた「大原則」を逸脱していないかの確認（ログベース）。
2. **銀行ステータス確認**: 現在の「銀行残高」および「買付余力」のリアルタイム表示。
3. **ポートフォリオ確認**: 株・投資信託の総額、保有量、保有割合（円グラフ等）、評価額、評価損益。
   - **ポジション情報の詳細表示**: 投資信託および長期運用（コア・サテライト）について、現在保有している具体的な銘柄名、保有口数/株数、取得単価、現在値、個別の評価損益を一覧表示（ポジション照会）できること。
4. **一覧性**: 上記2と3の情報を、「1スクロール以内（メインダッシュボード画面内）」で完全に把握できるUI。
5. **AI分析履歴の確認**: AIがどの銘柄を分析し、マクロ環境・ファンダメンタルズ等をどう評価し、どのような結果（買い/売り/待機）に至ったかをタイムライン等で確認可能。
6. **取引結果の確認**: 実行された取引ごとの利益・損失（金額および割合）。
7. **損失理由の確認**: 個別トレードで損失が発生した場合、ログから抽出された「何故損失になったか」の明確な理由表示。
9. **対象株式の個別照会**: ユーザー指定銘柄、または取引中銘柄の各種情報（時系列データ、指標）の参照。
10. **多角的な時間軸分析**: 各資産（銀行・投信・長期・短期）について、「日次（騰落・振替実績）」「月次（資産クラス別積み上げ成長曲線）」「年次（年間利回りと生活防衛費の安定性）」の3つの時間軸で推移を表示。
11. **短期トレード・イントラデイ詳細**: 09:15〜14:50の間、分単位で記録された含み損益＋実現損益の合計推移を可視化。VWAP乖離率やレジーム判定フラグのオーバーレイ表示を含む。
12. **レポート・アルバム機能**: 過去の日次・月次・年次の運用レポートをカレンダーやリスト形式で蓄積・一覧表示し、過去の成績やAIによる改善提案（振り返り）をいつでも閲覧できる専用ページ。

## 4. 自動トレード ロジック要件

本システムは、性質の異なる「パッシブコア（投資信託）」「長期運用（株式）」「短期運用（VWAP）」の3層構造の取引エンジンを連携・稼働させます。

### 4.1 長期運用・マルチバガー・クオンツ（Gemma 2利用）
将来のキャッシュフロー（FCF）の成長を狙うエンジン。総資産の大部分を割り当て、安全性を重視する「コア」枠と、一攫千金を狙う「サテライト」枠の二層構造。

- **コア・エンジン (資産の70〜80%)**: 堅実な複利。
- **サテライト・エンジン (資産の20〜30%)**: 流動性を厳格に制限しつつ、圧倒的財務を持つ10倍株候補を大底圏で取得。
- **Gemma 2の役割**: マクロ環境（為替、金利）や国策を考慮してスコア補正を行う「時間のかかる深遠な意思決定」。

### 4.2 短期トレード（ローカルLLM活用）※一部非公開
機関投資家の行動原理に起因する「市場構造の歪み」を突き、有意なシグナルを捉えるエンジン。
単純なテクニカル指標の組み合わせによるノイズを排し、強固なエッジを構築する。

- **ローカルLLMの役割 (Gemma2 / Codestral / Llama)**:
  - アクティブ時間を外れた事前準備フェーズや事後レビュー、コード生成のほか、市場構造やマクロデータの解釈、トレードログの定性分析をセキュアな環境で実施。
  - 最新の市況ニュースやセンチメントを構造化し、レジーム判定の補助を行う。
  - **レポート生成と改善点分析 (If-Then分析)**: 取引終了後、個別銘柄の分析結果に加え、「どこでエントリー/エグジットすればより利益が出せたか、損失を少なくできたか」といったIf-Thenシナリオ（反省点と改善案）を自動生成し、レポート・アルバムに格納する。
- **稼働時間とリスク管理**:
  - **計算アルゴリズムの絶対化**: アクティブ時間中（09:15〜14:30）はAIの推論を完全にシャットダウンし、高頻度で異常検知ベースの自動執行のみに限定。
  - **オーバーナイトリスクの排除**: 14:50に必ず全ポジションの強制決済を行い翌日に持ち越さず、ドローダウンの最小化を最優先する。

## 5. エラーリカバリー・リスク管理
- **キルスイッチ**: 意図しない大規模ドローダウン時（例: 資産の3%）に自動で発注をブロック。
- **積立金保護**: 生活防衛費を下回る銀行余額での発注ロック。
- **全層ロギング**: データ収集、モデル推論、API発注、それぞれで障害時のエラー箇所（Component）を明示して記録しDiscordへ通知。
